# 04 运行手册 — 服务器跑代码的【唯一标准流程】

> **为什么有这份**：避免"每开新窗口、一跑代码就要重新讨论，且每个窗口给的命令方式都不一样"（user 2026-06-13 提出的真实摩擦）。
> **硬规矩**：任何窗口给 user 的运行命令**必须**与本文件 §2 一致；要改流程**先改这里、再给命令**，不许临时另造一套。
> **维护型文档**（不是日志）：跑的"目标"变了（mini→全量→Phase 2/3 SAC）就**更新 §2**；§0/§1（流程与约定）基本不变。

---

## §0 标准 5 步（不随窗口变）

1. **传/更新代码**：只传改过的文件（多数时候就 `代码/run_step4e.py` 一个）。
2. **自检代码是最新**：`grep -c anneal_ent_coef 代码/run_step4e.py` → 出 **2**=最新；**0**=旧版，重传 `run_step4e.py`。
3. **（首次）建持久场景缓存**：见 §3，设一次、以后不再联网重下。
4. **screen 后台跑**：`screen -dmS step4e bash run_strided.sh`（断 ssh 不停）。
5. **回传结果**：把 `结果/table3_*.txt` 发回 Claude 核对是否过通过门。

## §1 固定约定（永远不变，新窗口照搬）

- **目录结构**：服务器上 `代码/`（含 run_step4e.py / launch_step4e.sh / trb_env/）与 `run_strided.sh` **并排**放同一层（如 `~/trb/`）。结果自动落 `~/trb/结果/`。
  - ⚠️ `run_strided.sh` 要放 `代码/` 的**旁边**、不是塞进 `代码/` 里（脚本按 `代码/xxx` 相对找文件）。
- **python**：用镜像 base 的 `python`（走 PATH）。代码强制 `device=cpu`（**吃 CPU 不吃 GPU**、别租 GPU、`03` D17）。
- **长训一律 screen 后台**（`run_in_background`/前台都会被会话暂停杀，实测多次）：
  - 起：`screen -dmS step4e bash run_strided.sh` ｜ 看：`screen -r step4e`（退出 `Ctrl+A` 再 `D`，任务不停）｜ 列：`screen -ls`
  - **被杀/断网就再敲一次同一条启动命令**——已完成的 (方,种子) 自动跳过、续跑。
- **场景**：脚本自动从 gitlab 下、且 `>1KB` 的自动跳过（缓存复用）。持久化见 §3。
- **服务器三件套**（已写进脚本，无需手动）：`OMP=1`（防 BLAS 线程超订 `03` L22）+ `device=cpu` + python 走 PATH。
- **定核（`03` L25 实测定论）**：自动 K 只是**保守下限**——on-policy PPO 单任务"采集↔更新"相位交替、占不满核，要**调大 K（并行任务数）盯 CPU% 加到 ~70%**（留 25-30% 余量最稳、别飙 95%+ 过头）。**K 可任意正整数（3/5/6/7 都行、不限 2 的倍数）**。**当前这台 32vCPU AutoDL 机实测最优 = K=6**（CPU 68-76%、~2.5× 提速、mini ~5h）。换机器从小往上、照 CPU% 重调。调 K：直接编辑 run_strided.sh 里 `K=6; NENVS=8` 那行的 K 数字（K 与 NENVS 同行、别用旧 `sed '/^K=/...'` 会误删 NENVS）→ 停掉重起（`03` L25：改 K 重起会让没跑完的任务从头来，调到位就定）。
